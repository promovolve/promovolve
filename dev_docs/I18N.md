# Dashboard i18n (English + 日本語)

Scope: the Go-template dashboard (`platform/templates/` + handler-emitted
strings in `platform/internal/handler/`). The banner runtime localizes by
*content* (Japanese creative → 「閉じる」), the designer UI is
English-only for now, and the help guides stay English — all out of scope
here. This file documents how translation works and the rules for adding
strings without breaking CI or, worse, production data.

## The model in one paragraph

Translation is applied **server-side, at template render time**. There is
one catalog, `platform/internal/i18n/ja.go` — a Go map where **the English
text is the key**: `"Campaigns": "キャンペーン"`. Each page template is
parsed once per language with a `t` func bound into its funcMap; the
request's language picks the pre-parsed variant, and every `{{t "..."}}`
is substituted during render. The HTML that leaves the server is already
in the reader's language — there is no client-side dictionary. English
needs no catalog: a key missing from `ja.go` falls through as itself, so
partially translated surfaces degrade to English, never to a broken page.

Language resolution (`i18n.Resolve`): the user's explicit preference
(`platform_users.locale`, set on /account/preferences — Auto/English/日本語)
wins; `""` (Auto) falls back to the browser's `Accept-Language`; default
English. Signed-out pages negotiate from the browser alone. JSON error
toasts (`h.jsonErrorT`) follow the browser, since most fire pre-login.

## Adding a user-visible string

### In a template

```html
<h1>{{t "Campaigns"}}</h1>
<p>{{t "Showing %d of %d sites" .Shown .Total}}</p>
<button onclick="return confirm('{{t "Delete campaign %s? Reports are kept." .Name}}')">
```

- The English text is the key — write it verbatim, punctuation and all.
- Interpolation is printf-style; `%d` for ints, `%s` for strings. The
  *translated* string is the format, so Japanese may reorder with indexed
  verbs: key `"Remove %s from %s?"` → ja `"%[2]sから%[1]sを削除しますか？"`.
- Pipeline form works too: `(t "Campaign")` as a dict arg, and
  `{{$x = t "..."}}` assignments. The drift scanner sees all three forms.
- Then **add the Japanese value to `ja.go` in the same commit** — the
  drift test fails CI otherwise (see below).

### In Go

```go
Error: i18n.T(h.lang(r, user), "could not save preferences — try again")
```

`h.lang(r, user)` resolves the request language (nil user = browser).
Page `Title:` fields are the exception: leave them plain English literals
— `renderStatus` translates titles centrally, and the drift scanner
counts `Title: "..."` literals as used keys. Composed titles
(`"Billing · " + x`) must instead be printf keys at the call site:
`i18n.T(lang, "Billing · %s", x)`.

### Enum values shown to users

Render through `t` dynamically — `{{t .Status}}` — and register each
possible value in the `dynamicKeys` allowlist in
`platform/i18n_drift_test.go` (the literal scanner can't see dynamic
lookups; the allowlist entry documents where the lookup happens). English
output is unchanged because the key falls through.

## The four hard rules

1. **Never translate an identifier.** Translation bakes into whatever the
   template emits — including JavaScript source. If a script *stores or
   compares* the string (form values, role names, keys in localStorage),
   wrapping it in `{{t}}` corrupts data, not just display. This shipped a
   real bug: the wizard's brand-kit color names ("Background", "Text"…)
   are looked up by name to derive page backgrounds; translating them
   made backgrounds transparent in Japanese sessions (`f2ae757`).
   Display strings and identifiers must never share a variable — if a
   value needs both, keep the identifier English and translate only at
   the display site.
2. **Never put `{{t}}` inside a JS backtick template literal** — Go ≥1.21
   refuses to parse it. Hoist a `const I18N = { save: '{{t "Save"}}' }`
   at the top of the script and reference `I18N.save` (including inside
   backticks via `${I18N.save}`).
3. **Japanese catalog values must not contain `'` `"` `\` backtick or
   `</`.** Alpine attribute expressions (`x-text="'...'"`) get only HTML
   escaping — a quote in the value breaks the expression at runtime with
   no template error. Use 「」. The drift test enforces this. (Plain
   `on*` attributes like `confirm('...')` are safe — html/template
   JS-escapes those contexts — but the catalog rule applies everywhere
   because values don't know where they'll be used.)
4. **Strings substituted by client JS keep literal verbs.** A few keys
   (`Page %d`, `Creative - %s`) are filled by `.replace('%d', …)` in the
   browser, not by Go's Sprintf — their ja values must keep the token
   exactly once and must NOT use indexed `%[n]d` forms.

## The drift guard (CI)

`platform/i18n_drift_test.go` keeps templates, Go, and the catalog in
lockstep:

- every literal key used in templates (`{{t "…"}}`, `(t "…"`, `= t "…"`)
  or Go (`i18n.T`, `jsonErrorT`, `Title:` literals) must exist in `ja.go`;
- every `ja.go` entry must be used somewhere (or be registered in
  `dynamicKeys` with a note saying where the dynamic lookup lives);
- no forbidden characters in values (rule 3);
- printf verbs in the value must match the key (indexed forms count).

A bilingual render matrix (`internal/handler/i18n_render_test.go` plus
the per-page render tests, which each loop `{en, ja}`) executes every
template in both languages, so parse errors and format-verb mismatches in
either language fail CI too.

## Translation style

- Tone: です・ます調 throughout.
- Terminology follows the book's glossary (`book/src-ja/glossary.md`):
  インプレッション, クリエイティブ, フロア, フィル率, ドッグイヤー,
  ペーシング, 精算, クリアリング価格. Don't invent a second term for a
  concept the book already names.
- One key = one translation everywhere. If one English word needs two
  different translations in two contexts ("Revenue" as a bid-weight label
  vs a table header), either pick the value that reads acceptably in
  both, or reword one English source so the keys differ.
- No plural machinery: `%d campaigns` is the whole key. English shows
  "1 campaigns" in the singular; we accepted that over a plural system.

## Adding a language (future)

`Resolve` and `T` are the only places that know the language list, and
`T(lang, key, args)` is the abstraction boundary — every one of the
~1,000 call sites talks only to that signature, so whatever the
internals become, no template or handler sweep ever happens again.

Mechanically, a new language is: a new catalog map + a branch in `T`, a
subtag in `Resolve`, an `<option>` on the preferences page, extending
the drift test to the new catalog, and — the actual work — the
translations themselves (anchored to the book glossary, reviewed like
code). The template cache scales per language automatically (~35 parsed
templates per language, in memory).

### Migration policy: when the homegrown core stops being enough

The current zero-dependency design was shaped by Japanese, and CJK is
its sweet spot: no plural forms, no gender, the same value rules.
Decision on record (2026-07-17):

- **Adding a CJK language (zh, ko)** → extend the current system as-is.
  The plumbing is an hour; the translation is the work.
- **First plural-bearing language (es, fr, de — especially ru/pl)** →
  adopt `golang.org/x/text/message` (or go-i18n's CLDR plural
  selectors) *inside* `internal/i18n`, behind the same `T` signature.
  Our "no plural machinery" decision ("1 campaigns") is tolerable in
  English and moot in CJK, but wrong in Spanish and unworkable in
  Russian (three forms). Keep English-as-key (it is also `x/text`'s
  native keying) and port the drift guard to read the new catalog
  format — the guard is our real asset and is library-independent.
- **First external translator** → the Go-source catalog becomes a
  bottleneck; add a small generator round-tripping `ja.go` ↔ JSON/PO
  for handoff (or move catalogs to embedded files outright). Until
  then, translations reviewed in code diffs are a feature, not a gap.

`Resolve` already honors Accept-Language q-values (weights, `q=0`
exclusion, primary-subtag collapse), so negotiation quality does not
degrade as the list grows; `language.Matcher` becomes worth importing
only alongside the `x/text` move above.

## What is deliberately not localized

Dates keep their current layouts (ISO-ish, locale-neutral); currency is
`$` throughout; IANA timezone names, emails, domains, and IDs stay raw;
core-API-origin strings (layout template names, Gemini arc tags) would
need API-side localization. The month name in the shared report calendar
*does* follow the page language via `document.documentElement.lang`.
